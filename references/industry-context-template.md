# 行业上下文模板

当需要把 `tob-sales-coach` 适配到某个具体行业、产品、服务、公司或团队时，使用本文件。

## 首次配置与持续深挖流程

用户说刚下载、第一次使用，或当前对话里没有已建立的行业上下文时，先检查当前工作区是否存在：

```text
./tob-sales-context/onboarding-state.md
./tob-sales-context/industry-context.md
```

判断规则：
- 如果 `industry-context.md` 已存在，视为已经配置完成；不要再询问是否配置，默认按该上下文处理后续客户。
- 如果 `onboarding-state.md` 显示用户首次选择继续使用通用版，后续不要再次主动询问配置，除非用户主动说“先配置一下/我要配置/更新行业信息”。
- 如果两个文件都不存在，视为没有配置记录；不管用户第一条消息是客户线索、通话记录、报价问题还是普通问题，都先暂停处理原始请求，让用户选择“先配置”或“先继续”。

首次使用时先让用户选择：

> 这个通用 ToB 销售教练可以直接使用：你发任意 B2B 线索、通话记录、报价问题或谈判场景，我都能先按通用销售逻辑帮你判断。  
> 但如果我先了解你的行业、产品、客户、价格、交付边界和常见异议，后面的客户判断、话术、报价、复盘会更贴近你的真实业务。  
> 你要先配置，还是先继续？

如果当前环境提供 `request_user_input`，用它展示 1 个选择问题，给 2 个互斥选项：
- 先配置（推荐）：先上传公司资料，再根据资料缺口持续追问；配置不会自动结束，需要用户明确说“结束”才收口。
- 先继续：不配置，直接处理当前任务，后续不再主动询问配置。

如果 `request_user_input` 不可用，用普通聊天问题让用户回复选项编号：

```markdown
你要先配置，还是先继续？

1. 先配置（推荐）：先上传公司资料，再根据资料缺口持续追问；配置不会自动结束，需要你明确说“结束”才收口。
2. 先继续：不配置，先直接处理你刚才发的内容，后续我不再主动问你是否配置。
```

用户回答“要/可以/先配置/深度配置”时，先进入资料收集，再进入问诊。

用户回答“不要/先不用/直接分析/先处理客户”时：
- 不再追问配置，直接按通用版本处理用户原始销售请求。
- 如果可以写入当前工作区，创建或更新 `./tob-sales-context/onboarding-state.md`，记录“用户首次选择继续使用通用版；后续不主动询问配置，除非用户主动要求配置”。
- 如果不能写入文件，则只在当前对话中记住这个状态。

用户没有回答是否配置时，不要继续输出客户分析；先等待用户选择。

建议的 `onboarding-state.md` 格式：

```markdown
# ToB Sales Coach Onboarding State

- status: continue-general
- note: 用户首次使用时选择继续使用通用版；后续不主动询问配置，除非用户主动要求配置。
```

完成配置后的 `industry-context.md` 应包含最终行业上下文、关键销售假设、价格和交付边界、常见异议、让步规则和仍需确认的问题。

## 深度配置流程

用户确认需要配置时，先让他上传或放置公司资料：

> 可以。你先把公司可以提供的资料上传，或放到当前工作区的 `./tob-sales-context/` 目录。  
> 如果暂时没有资料，也可以直接告诉我“没有资料，直接问”，我会直接开始问诊。配置过程中我会持续深挖，直到你明确说“结束”才会收口并生成配置总结。

优先读取资料：
- 如果 `./tob-sales-context/` 已存在且有文件，先递归读取可用资料，提取关键信息，再根据缺口决定需要追问的问题。
- 如果用户上传了文件，先读取文件，再生成问题。
- 如果用户明确说没有资料，直接进入问诊。
- 如果资料很多，先提取和销售相关的信息：公司业务、产品、价格、案例、合同、竞品、销售话术、客户记录。

读取资料后，先输出三类信息：
- 已确认资料：文件中直接出现的信息。
- 销售推断：基于材料可以合理推断的信息。
- 仍需确认：影响销售建议但资料缺失的信息。

然后只追问资料里缺失的关键信息；不要重复询问文件中已经确认的信息。

## 可点选提问策略

如果当前环境提供 `request_user_input`，必须用它展示可选选项。每次最多问 1 个问题，给 2-3 个互斥选项，并把推荐项放第一位。

如果 `request_user_input` 不可用，不要退回一题一题编号问答。改为生成本地 HTML 深挖表单：
- 在当前工作区创建 `./tob-sales-context/deep-dive-form.html`。
- 表单根据已读取资料的缺口生成一组问题，一次性收集信息。
- 表单优先使用单选、多选、下拉框和文本框；选择题选项必须互斥时用单选，不互斥时用多选。
- 每个选择题给 2-3 个高质量选项；推荐项放第一位并标注“推荐”。
- 页面底部必须提供“生成分析输入”按钮。
- 点击后输出一段可复制内容，包含所有选择、文本补充和仍不确定的问题；用户把这段内容粘贴回 Codex 后，再继续分析或生成 `industry-context.md`。
- 表单问题数量不固定，但要聚焦关键缺口，不要把题库完整搬进去。

只有在无法生成或打开 HTML 时，才使用普通聊天问题。此时仍优先给 2-3 个选项让用户选择，不要只让用户自由输入文字。

问诊时：
- 不设固定问题数量，不设固定组数；用户不说结束，就持续深挖。
- 使用 `request_user_input` 时，每次只展示 1 道题和 2-3 个互斥选项，不要一次展示多道题。
- 使用本地 HTML 表单时，一次性展示一组问题；用户提交表单结果后，再根据新缺口决定是否继续生成下一组表单。
- 用户回答后，先把答案纳入上下文，再根据当前最大信息缺口生成下一题。
- 只有用户明确说“结束/够了/完成/先到这里/停止提问/不用再问了”时，才停止追问并进入总结。
- 如果用户只是回答当前问题或提交表单结果，没有表达结束意图，必须继续深挖下一题或下一组表单。
- 问题顺序由资料缺口决定，优先追问会影响销售判断和话术质量的高价值信息：产品/服务、目标客户、ICP、决策链、价格模型、成交周期、痛点、触发事件、竞品、交付边界、让步规则、常见异议、成功案例、销售团队短板。
- 使用 `request_user_input` 或普通聊天兜底时，顶部必须标注进度：`配置问诊 · 第 N 题`。
- 使用 `request_user_input` 时，用户完成当前题作答后，自动进入下一题；不要额外询问“是否继续”，除非用户主动表达想结束。
- 用户说结束后，统一汇总所有选择、表单输入和文本补充。
- 用户要求“简版”时，也不设固定题量；使用 `request_user_input` 时减少单题数量，使用 HTML 时减少表单问题数量，但仍持续深挖到用户说结束。
- 用户回答不完整时，基于已知信息继续，并列出最重要的未知项。
- 用户说结束后，汇总行业上下文，并形成一版可工作的销售手册假设；如果仍有关键缺口，要列为“后续可补充”，不要继续追问。
- 如果可以写入当前工作区，创建或更新 `./tob-sales-context/industry-context.md`，以后默认按该上下文提供建议，不再询问是否配置。
- 问诊题库只是备用框架；如果已经读取了资料，要基于资料缺口智能改写问题，不要机械照抄题库。

只有在无法生成或打开 HTML 时，普通聊天问题按以下格式：

```markdown
配置问诊 · 第1题
问题：公司主要是做什么业务？

1. 产品生产制造（推荐）
2. 线上电商服务
3. 线下实体门店
```

每次仅输出单题和 2-3 个互斥选项，无多余解释文字。不要用 A/B/C 字母选项。

自动流转示例：

```markdown
配置问诊 · 第1题
问题：[问题 1]

1. 选项 1（推荐）
2. 选项 2
3. 选项 3
```

用户提交后，立刻输出：

```markdown
配置问诊 · 第2题
问题：[问题 2]

1. 选项 1（推荐）
2. 选项 2
3. 选项 3
```

只要用户没有说结束，就继续输出下一题。不要在同一条消息里提前展示后续问题。

## 问诊题库

下面是备用问题池，不是固定问卷。根据资料缺口和用户上一题答案，从中选取或改写下一题；用户没有说结束就持续深挖，用户说结束就停止。

### 1. 公司和业务

1. 你们公司主要卖什么产品或服务？
2. 你们现在主要服务哪些行业或客户类型？
3. 你们目前更重视新客增长、老客复购、客单价提升，还是销售流程标准化？

### 2. 产品和交付

1. 你们的产品/服务是标准化、半定制，还是强定制？
2. 客户购买后，你们具体交付哪些内容？
3. 哪些事情是你们明确不交付、不承诺或必须客户自己负责的？

### 3. 目标客户和 ICP

1. 什么样的客户最适合你们？
2. 什么样的客户即使有需求也不适合接？
3. 你们判断客户质量时最看重规模、预算、行业、痛点、决策人，还是时间节点？

### 4. 决策人和影响人

1. 典型客户里谁最先接触你们？
2. 最终拍板人通常是谁？
3. 采购、财务、法务、技术、安全、业务使用者里，谁最容易影响成交？

### 5. 客户痛点

1. 客户通常因为什么问题来找你们？
2. 这些问题如果不解决，会造成什么业务损失或风险？
3. 客户最常用哪些原话描述这些问题？

### 6. 触发事件

1. 客户一般在什么节点开始考虑采购？
2. 是增长压力、降本增效、合规要求、系统替换、团队变化，还是老板推动更常见？
3. 哪些信号说明客户现在真的着急？

### 7. 价格和预算

1. 你们常见客单价或报价区间是多少？
2. 你们是按项目、订阅、人数、用量、年框、阶段，还是其他方式收费？
3. 客户最常见的预算异议是什么？

### 8. 销售周期和阶段

1. 一个客户从首次接触到成交通常多久？
2. 中间通常会经历哪些关键节点？
3. 哪个阶段最容易卡住或丢单？

### 9. 竞争和替代方案

1. 客户通常会拿你们和哪些竞品比较？
2. 客户如果不买你们，一般会用什么替代方案？
3. 你们最有优势和最弱势的比较点分别是什么？

### 10. 价值主张

1. 你们最核心的价值主张是什么？
2. 客户买你们之后，最应该看到哪些变化？
3. 你们有哪些说服力最强的案例、数据或故事？

### 11. 常见异议

1. 客户最常说的 3 个反对意见是什么？
2. 哪些异议是真问题，哪些只是拖延或压价？
3. 你们现在通常怎么回应这些异议？

### 12. 让步和边界

1. 哪些条件可以作为商务让步交换？
2. 哪些价格、范围、服务、合同条款绝不能轻易让？
3. 如果客户要求折扣，你们希望换回什么确定性？

### 13. 资料和内容资产

1. 你们现在有哪些销售材料：官网、PPT、案例、报价单、合同、FAQ、录音、聊天记录？
2. 哪些材料最能帮助客户理解价值？
3. 哪些材料不能外发，只能内部参考？

### 14. 成交和成功标准

1. 客户内部通常用什么标准判断是否值得买？
2. 成交后客户如何判断项目成功？
3. 你们希望销售在签约前确认哪些成功标准？

### 15. 销售团队和训练目标

1. 你希望这个 skill 主要帮你做线索判断、话术、复盘、谈判，还是销售管理？
2. 你或团队现在最常见的销售弱点是什么？
3. 你希望 AI 每次输出更像销售主管、销售教练，还是一线销售搭档？

## 产品和价值映射

拿到上下文后，建立这些映射：

- 产品功能 -> 客户业务问题。
- 服务模块 -> 客户成功指标。
- 交付边界 -> 风险控制。
- 价格组成 -> 价值和投入。
- 实施要求 -> 客户侧配合事项。
- 案例 -> 相似客户处境，而不只是行业相似。

## 行业包模式

如果用户想沉淀成可复用的垂直行业版本，可以新增类似文件：

- `references/saas-context.md`
- `references/consulting-context.md`
- `references/training-context.md`
- `references/industrial-equipment-context.md`
- `references/marketing-services-context.md`

每个垂直文件应包含：
- ICP 和非 ICP。
- 买方画像。
- 需求挖掘问题。
- 演示/demo 或 proposal 规则。
- 价格模型和边界。
- 常见异议。
- 采购/法务/安全问题。
- 案例和故事角度。
- 红线。

垂直文件要和通用手册分开，不要把某个行业的黑话污染到通用 ToB 规则里。

## 上下文缺失时

使用通用 ToB 逻辑：
- 推产品前先问业务问题。
- 提 proposal 前先确认决策链。
- 谈价格前先确认预算路径。
- 定范围前先确认成功标准。
- 做预测前先确认采购/法务流程。

如果建议基于通用 ToB 假设，而不是已确认行业上下文，要明确说明。
